## Custom Project

### Dataset

Describe the dataset used by your Kafka producer.

- **Name of the dataset file:** The streaming pipeline processes a sales transaction dataset stored in the data folder under the name sales.csv.
- **Kind of records it contains:** This file contains real-time retail commerce records, representing individual customer transactions across multiple business areas.
- **Fields included in each record:** Each row in the dataset includes specific fields such as a unique order identifier (order_id), a product identifier (product_id), the quantity of items purchased (quantity), the base unit price (price), and a geographic region identifier (region_id).
- **Dataset modifications:** The original sales dataset structure was preserved to maintain consistency with the baseline enterprise data contract definitions while testing pipeline architectural modifications.

### Kafka Messages

Describe the messages sent through Kafka.

- **What your producer sends:** The Kafka producer continuously reads the raw records from the dataset and transmits them as live message payloads serialized in a standard text-based JSON data exchange format.
- **Kafka topic used:** These messages are broadcasted across a dedicated network stream topic specified within the configuration settings as the sales topic.
- **Message key used:** Each transmitted message utilizes the product_id or order_id as the message routing key to guarantee that messages from the same operational categories are ordered sequentially within their network partitions.
- **Message field changes:** The primary message fields from the original dataset were kept intact during transmission to allow the consumer engine to evaluate a complete data contract profile during live consumption.

### Consumer Processing

Describe what your consumer receives and does with each message.

- **What your consumer receives from Kafka:** The consumer application establishes an active connection to the message broker and receives a continuous sequence of live transactional JSON message packets from the topic stream.
- **Number of messages consumed:** The engine is configured to process messages up to a maximum limit of 1000 messages or until a specific inactivity timeout threshold of 10 seconds is reached with no new incoming stream events.
- **What it logs or prints:** For each received message, the consumer logs operational status updates, validation checks, and calculated financial summaries directly to the system console.
- **Writing records to a CSV file:** The consumer does not write records to a traditional comma-separated CSV file; instead, it has been optimized to append rows directly into a JSON Lines (.jsonl) text stream file for durable stream storage.
- **Processing and filtering selected fields:** It filters and processes selected fields to verify compliance with strict data validation contracts before appending them. Specifically, it confirms the presence of required fields, executes live regional tax lookups, computes derived metrics including subtotals, tax values, and total order amounts, and aggregates cumulative processing metrics for the entire active session.

### Experiments

Describe the small technical changes you made.

- **Phase 4 change (Technical Modification):** The technical modification introduced for the architecture upgrade involved replacing the standard multi-column plain text comma-separated output file pattern with an optimized line-by-line JSON Lines (.jsonl) streaming storage structure inside the personal consumer module kafka_consumer_mahammad.py.
- **Phase 5 application (New Application):** For the application phase, a custom analytical visual charting module named chart_generation_hajiyev.py was successfully integrated into the system package to automatically connect to the local relational DuckDB database, extract historical stream aggregates, and render standalone trend plots.

### Results

Describe what happened when you ran the producer and consumer.

When executing the data pipeline, the producer successfully loaded the source transactions and streamed individual message blocks sequentially without experiencing network drops or latency spikes. Simultaneously, the custom consumer intercepted each message packet, executed real-time validation protocols, and routed approved transactions smoothly into both the relational local DuckDB storage engine and the newly configured streaming .jsonl text file. Running the visualization script pulled data records directly from the database and produced a clean distribution chart illustrating message density per region with zero processing exceptions.

### Interpretation

Explain what the Kafka streaming workflow showed you.

- **What changed from the original example:** The pipeline moved away from raw flat CSV tracking and established a more resilient streaming data format by implementing a sequential JSON Lines file combined with automated backend chart generation using Python module lookups.
- **What you learned from watching messages move through Kafka:** The streaming architecture demonstrated the efficiency and reliability of decoupling message generation from long-term database storage operations. This experiment showed that streaming text files allow systems to capture active records line-by-line in an append-only configuration, protecting the system from file structural corruption if a power loss or consumer crash occurs mid-stream.
- **What the stream could tell a business or organization:** For a business or organization, this workflow offers direct operational intelligence by highlighting real-time regional market performance, showing exactly which locations are handling the highest transactional volume.
- **Business intelligence gained from consumed messages:** This stream provides immediate data on revenue metrics and regional activity patterns, allowing supply chain managers to adjust inventory distributions instantly based on live consumer demand rather than waiting for end-of-day batch processing updates.
